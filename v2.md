## 🎯 Visão Geral

A versão 3.0 reformula completamente o motor de extração para uma abordagem **determinística, baseada em grafos ortogonais** (↑ ↓ ← →), com:

* Representação de cada documento como **grafo de TextUnits (TUs)** conectados ortogonalmente;
* Identificação de **padrões de estilo** e **componentes de estilo** (TUs com a mesma formatação visual e proximidade);
* Classificação **não probabilística** dos TUs em papéis (`HEADER`, `LABEL`, `VALUE`) por **regras universais**;
* **Propagação de papéis** entre TUs de mesmo estilo e agrupamento;
* **Busca ortogonal e determinística** de pares `LABEL→VALUE`, guiada por **critérios de Pareto** (estrutura, estilo, tipo e léxico) — sem scores nem pesos arbitrários;
* **LLM Chooser** usado apenas quando as regras abstêm (empate real ou ausência de âncora textual), com cache e política baseada na **“temperatura” do documento**;
* **Memória de invariantes** que aprende e reutiliza padrões de layout, vocabulário, direções, offsets e shapes de valor.

Essa versão elimina completamente qualquer cálculo de “score” contínuo, substituindo-o por **decisões lógicas, hierarquias fixas e dominância Pareto**.
O objetivo é permitir **100% de interpretabilidade**, **reprodutibilidade** e **aprendizado progressivo** com cada novo documento processado.

## 🧠 Arquitetura Implementada

### 1. Núcleo conceitual

| Conceito                   | Descrição                                                                                                     |
| -------------------------- | ------------------------------------------------------------------------------------------------------------- |
| **TextUnit (TU)**          | Unidade mínima de texto homogênea em estilo e espaçamento.                                                    |
| **StyleSignature (S)**     | Vetor discreto (fonte, tamanho bin, bold, italic, cor, caps, espaçamento).                                    |
| **Role**                   | Papel semântico atribuído ao TU: `HEADER`, `LABEL`, `VALUE` ou `OTHER`.                                       |
| **Graph G(V,E)**           | Grafo ortogonal com V=TUs e E=arestas direcionadas ↑↓←→.                                                      |
| **StyleComponent**         | Conjunto de TUs conectados ortogonalmente com mesmo `S`. Recebe papel único por prioridade.                   |
| **ExtractionSchema**       | Mapa `{field_key: field_description}`.                                                                        |
| **Lexicon(field)**         | Conjunto fixo de tokens obtidos de `key` + n-grams curtos da descrição.                                       |
| **TypeGate(field)**        | Função determinística que reconhece se texto é compatível com tipo (`date_like`, `money_like`, `text`, etc.). |
| **InvariantMemory(label)** | Base de conhecimento incremental que armazena padrões observados por tipo de documento.                       |

### 2. Fluxo lógico (alto nível)

```
PDF
 → TextUnits (TUs)
   → Style Signatures
     → Graph G(V,E) (ortogonal ↑ ↓ ← →)
       → Roles (HEADER/LABEL/VALUE)
         → Style Propagation
           → Schema Anchoring
             → LABEL→VALUE Pairing (Pareto + tie-breakers)
               → LLM Chooser (apenas se necessário)
                 → Output + Proof
                   → Invariant Memory Update
```

## ⚙️ Pipeline Detalhado

### 1️⃣ Extração e normalização de Text Units

* Cada página do PDF é processada para gerar blocos elementares (TUs).
* Campos armazenados: texto limpo, bbox normalizado, tipografia (`font`, `size`, `bold`, `italic`, `color`), alinhamento horizontal, espaçamentos médios intra-token e inter-linha.
* Textos são transliterados (`á`→`a`) para simplificar matching.

### 2️⃣ Cálculo de Style Signatures

Cada TU recebe uma assinatura discreta:

```
S(TU) = (font_family_id, font_size_bin, bold, italic, color_cluster,
         caps_ratio_bin, letter_spacing_bin)
```

Bins são definidos por quantis globais da página, garantindo portabilidade.
Dois TUs com `S` idêntico são considerados do mesmo estilo.

### 3️⃣ Modelo de espaçamento (limiares automáticos)

O sistema mede:

* distâncias horizontais entre TUs da mesma linha;
* distâncias verticais entre TUs da mesma coluna.

Define limiares por estatística resistente:

| Limiar          | Cálculo                           | Uso                   |
| --------------- | --------------------------------- | --------------------- |
| `τ_same_line`   | mediana(gap_inter_token) + 1·MAD  | conexão horizontal    |
| `τ_same_column` | mediana(gap_inter_line) + 1·MAD   | conexão vertical      |
| `τ_multiline`   | mediana(gap_inter_line) + 0.5·MAD | detecção de multiline |

### 4️⃣ Construção do Grafo Ortogonal

* Arestas **↑ ↓ ← →** ligam TUs adjacentes segundo `τ_same_line` e `τ_same_column`.
* Nenhuma diagonal é criada.
* Cada TU pode ter múltiplos ↓ (caso de labels largos com várias linhas de valores).

## 🧩 Componentes do Sistema

### **src/graph/style_signature.py**

* Função de geração de `S(TU)` e agrupamento de TUs em `StyleComponents`.
* Clustering discreto de fontes, cores, tamanhos e espaçamentos.
* Interface:

  * `compute_style_signature(tu) → tuple`
  * `group_by_style(graph) → list[StyleComponent]`

### **src/graph/roles_rules.py**

* Define e aplica todas as regras R-H*, R-L*, R-V*.
* Propaga papéis por componente de estilo (`HEADER > LABEL > VALUE`).
* Corrige inconsistências (ex.: VALUE com separador → LABEL).
* Interface:

  * `assign_roles(graph, schema_lexicons) → graph with roles`

### **src/graph/orthogonal_edges.py**

* Cria o grafo ortogonal com base em `τ_same_line` e `τ_same_column`.
* Interface:

  * `build_orthogonal_graph(text_units) → Graph`

## 🧱 Classificação de Papéis e Propagação de Estilo

### HEADER

* Fonte ≥ quantil 90 da seção;
* Linha centralizada ou isolada;
* TU pai com ≥2 filhos ↓ de estilos diferentes.

### LABEL

* Termina com separador (`:`, `-`, `.`, `/`);
* ≤3 tokens, sem dígitos;
* Vizinho (→ ou ↓) passa type-gate;
* OU contém token do `Lexicon(field)`.

### VALUE

* Passa `TypeGate(field)` do schema atual;
* Está (→ ou ↓) de um LABEL;
* Não termina com separador.

**Propagação de estilo (E-1/E-2):**

* TUs conectados (↑↓←→) e com `S` idêntico compartilham o mesmo papel;
* O papel do grupo é decidido pela prioridade `HEADER > LABEL > VALUE`;
* Grupos de mesmo estilo se tornam **componentes de papel**.

## 🔗 Emparelhamento Determinístico LABEL→VALUE

### Estratégia

Para cada campo do schema:

1. Encontrar **LABELs candidatos**:

   * Categoria A: contém token exato do `Lexicon(field)`;
   * Categoria B: LABELs genéricos da região;
   * Categoria C: fallback curto (sem dígito).

2. Percorrer o grafo ortogonal em **ordem de direção fixa**:

   * Ordem: `→`, depois `↓`, depois `↑`, depois `←`;
   * BFS restrita (1–3 saltos) no mesmo componente de seção.

3. Coletar todos os nós que:

   * Passam o **TypeGate** do campo;
   * Não compartilham estilo com o LABEL;
   * Não violam ordem de leitura (“no-crossing”).

4. Aplicar **Filtro de Pareto** (ver abaixo).

## ⚖️ Seleção por Pareto e Tie-Breakers

### Critérios de Pareto (mesmo peso)

| Critério          | Descrição                                                              | Tipo   |
| ----------------- | ---------------------------------------------------------------------- | ------ |
| **C1 Estrutural** | Menos saltos; mesma linha/coluna; sem cruzamento.                      | lógico |
| **C2 Estilo**     | Estilo diferente do label; coerente com outros valores do mesmo campo. | lógico |
| **C3 Léxico**     | Enum: pertence ao conjunto; Texto: ≥4 letras e sem separador.          | lógico |
| **C4 Tipo**       | Passa o type-gate do campo.                                            | lógico |

Eliminamos candidatos dominados. O conjunto final contém apenas os **não-dominados**.

### Tie-breakers determinísticos

1. Direção preferida (`→` > `↓` > `↑` > `←`);
2. Menos saltos;
3. Menor distância Manhattan (Δx + Δy);
4. Menor `line_index`.

Persistindo empate → LLM Chooser.

## 🤖 LLM Chooser Inteligente

### Quando é acionado

* Empate real após Pareto + tie-breakers;
* Ausência de LABEL e múltiplos candidatos canônicos;
* LABEL textual desconhecido (não presente em memória nem lexicon).

### Entrada

* Até 3 pares `(label_text?, value_text, direção, vizinhos contextuais, estilo resumido)`;
* Informações do campo: `DocumentLabel`, `SchemaKey`, `SchemaDesc`, `enum_options`.

### Prompt

> “Escolha qual das opções representa melhor o campo `<SchemaKey>` descrito como `<SchemaDesc>`.
> Retorne apenas `{pick: idx|-1, why: "..."}`.”

### Pós-processamento

* Revalidação pelo **type-gate**;
* Cache por hash do subgrafo (nunca chama duas vezes para o mesmo caso);
* Integração com a memória: se confirmado, adiciona invariantes correspondentes.

## 🧬 Memória, Invariantes e Temperatura

### O que é salvo por `DocumentLabel`

| Tipo de Invariante       | Exemplo                             | Uso                        |
| ------------------------ | ----------------------------------- | -------------------------- |
| Vocabulário de LABEL     | “inscrição”, “seccional”, “nome”    | detecção rápida de campos  |
| Direção típica           | `→` ou `↓` predominante por campo   | priorização                |
| Offset                   | (dx, dy) médios em saltos discretos | validação estrutural       |
| StyleSignature dominante | bins de fonte e bold por campo      | validação de consistência  |
| Shape de valor           | regex inferida (`\d{6}`, `AA-9999`) | validação type-gate futura |

### Temperatura `T ∈ [0,1]`

Mede quão “quente” é o documento:

```
T = (# invariantes verificados) / (# invariantes aplicáveis)
```

**Uso:**

* T ≥ 0.8 → nenhuma chamada LLM (100% determinístico);
* 0.4 ≤ T < 0.8 → 1 chamada máxima (para desempate);
* T < 0.4 → até 3 chamadas (documento novo).

O sistema v3.0 substitui o modelo anterior (Grid+Graph+ScoreTuple) por um **Graph-Orthogonal Engine** interpretável e independente de pesos.
